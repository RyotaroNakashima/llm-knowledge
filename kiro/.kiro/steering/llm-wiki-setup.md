# LLM Wiki セットアップ

この steering は、Kiro で LLM Wiki をセットアップ・運用するためのガイド。

## 基本方針
Obsidian は optional。必須なのは、LLM エージェントが読める・検索できる・編集できる Markdown フォルダ。

Obsidian は人間用 UI として便利だが、会社環境で使えない場合は Markdown-only でよい。

## セットアップ時に作るもの
対象プロジェクトを確認してから、必要最小限の構成を作る。

```text
AGENTS.md
index.md
00_受信箱/
10_情報源/
20_ノート/
30_意思決定/
40_プロジェクト/
80_テンプレート/
90_運用/
```

Obsidian を使う場合だけ、必要に応じて `.obsidian/` を作る。

```text
.obsidian/app.json
.obsidian/core-plugins.json
.obsidian/templates.json
.obsidian/workspace.json
```

初期セットアップでは community plugin を入れない。

## 必須ファイル
- `AGENTS.md`: LLM エージェントが作業前後に従う入口ルール。
- `index.md`: Wiki の入口。現在のテーマ、主要ノート、未解決事項を書く。
- `90_運用/運用マニュアル.md`: フォルダの役割、日常ワークフロー、ノート基準、保守方法を書く。
- `90_運用/LLM記憶プロトコル.md`: LLM が最初に読むもの、更新ルール、残す情報、残さない情報、作業後プロンプトを書く。
- `40_プロジェクト/<project-name>.md`: 現在の状態、重要な文脈、次の行動を書く。
- `30_意思決定/YYYY-MM-DD-LLM記憶にMarkdownを使う.md`: Markdown を使う理由、却下した案、結果を書く。

## AGENTS.md に入れること
`AGENTS.md` は短くする。毎回すべてのノートを読ませない。

必ず入れる内容:

- 作業前に読むもの: `index.md`、`90_運用/LLM記憶プロトコル.md`、関係ノート。
- 作業後の更新判断: 意思決定、古くなったノート、新しい運用ルール、失敗案、プロジェクト状態、今後も効くユーザー制約がある場合だけ更新。
- 更新先の使い分け: `00_受信箱/`、`10_情報源/`、`20_ノート/`、`30_意思決定/`、`40_プロジェクト/`、`90_運用/`。
- 作業ログ全文は保存しない。
- 必要な場合だけ短い作業メモを `00_受信箱/` に置く。
- 更新不要なら無理に Wiki を触らない。

## ノート基準
長期保存するノートには、できるだけ次を入れる。

```text
状態: 現行 | 下書き | 古い | 置き換え済み
最終更新: YYYY-MM-DD
信頼度: 低 | 中 | 高
```

次を分けて書く。

- 事実
- 判断
- 理由
- 却下した案
- 未解決事項
- 関連リンク

## 残す情報
残す:

- 意思決定と理由
- 避ける価値がある失敗案
- 再調査を減らすプロジェクト状態
- 再利用できる概念
- 今後の作業を導く未解決事項
- 将来のエージェントが守るべき運用ルール

残さない:

- チャット全文
- 明らかなコマンド出力
- 完了済みの一時的な作業計画
- 根拠のない曖昧な好み
- 日付や出典がない鮮度依存情報

## 作業後プロンプト
まとまった作業の後、必要な場合だけ使う。

```text
この作業から、将来の LLM エージェントや自分が再利用すべき長期記憶を抽出してください。

既存の Wiki を読み、必要なら既存ページを更新し、新規ページは最小限にしてください。
事実、判断、根拠、却下した案、未解決事項、次回の注意点を分けて書いてください。
古くなりそうな情報には日付と信頼度を付けてください。
チャット全文や一時的な作業ログは保存しないでください。
```

## 検証
セットアップや変更後は次を確認する。

- `find` または `rg --files` で期待ファイルがあること。
- 古い Markdown リンクが残っていないこと。
- `.obsidian/*.json` がある場合は JSON として正しいこと。
- 最終回答で、作成・更新した主要ファイルを短く伝えること。
